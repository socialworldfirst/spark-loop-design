The Spark Loop

A standing controller that keeps your spark pipe permanently staged: every run parked exactly at a human decision, never sitting on dead auto-work, with the bank and the run queue always topped up. It does not replace anything. It sequences the machine you already built.

Designed 9 June 2026 Reads: content_bank.json · runs/*/run.json Form: /loop self-paced on a work-hours heartbeat

01 What the loop sits on

Your pipe is already two layers. The loop is the missing third thing that moves work from the first into the second.

The bank holds candidate angles with a Status. The runs hold active productions with a stage and an open_gate. Four gates are amber (your call), everything between them is green (Claude's to run). The loop is the engine that pulls greenlit angles out of the lake and walks them through the machine, gate to gate.

02 What is already looped, and the one gap

Half the loop already exists as cron. The refill side is done. The drive side is the hole.

Today the bank fills itself every Monday, but nothing spends it. Greenlit angles sit in the lake until you personally open a /spark run and hand-walk it. The loop closes that: the lake fills on its own, and now it drains on its own too, stopping only at your four decisions.

03 The design: one controller, two engines

Not a fleet. One self-paced loop, woken by a work-hours heartbeat. The two /loop engines map cleanly onto the two speeds the pipe runs at.

Heartbeat · CronCreate

A scheduled task fires the controller every ~2h between 09:00 and 21:00. Dumb, reliable, off at night (no point staging boards at 3am). This is the steady tick that reconciles state to target.

Fast poll · ScheduleWakeup

While a research or script workflow is mid-flight, the controller self-schedules a ~270s re-check (stays inside the prompt cache window) to catch completion, then drops back to the slow heartbeat. Smart cadence only when something is actually moving.

Why not /loop until? Because there is no finish line. This is a standing reservoir controller, not a task that completes. So the form is self-paced (heartbeat + fast poll), and the goal is a level it holds, not a condition it ends on. The until form is the right shape only for a single run you want chased to done.

04 The goal it converges to

Drive gate-to-gate dead air to zero. Every run is always parked at a human decision, never mid-auto-stage. And keep two buffers full so you never hit a cold start.

B · bank buffer target ≈ 10

Greenlit, research-ready angles waiting in the lake. The weekly scan keeps this full; the controller escalates an extra /spark_scan only if it runs critically dry.

R · run buffer target ≈ 3

Live runs in-flight, each advanced to and parked at its next gate. If fewer than R runs have a gate open, the controller spawns a new one off the top-fit greenlit angle.

These two numbers are the whole policy. Everything the loop does each tick is in service of holding B and R at target while never crossing a gate it is not allowed to cross.

05 The per-tick reconcile

One tick is one pass of this list. It is idempotent: if there is nothing to do, it notifies no one and sleeps.

  1. Load state. Read _active.json, every runs/*/run.json, and content_bank.json. Build the picture: who is at a gate, who is on auto-work, how full are B and R.
  2. Drain auto-work.green Any run with open_gate=null and stage≠done: advance the next auto stage. Heavy stages (listen+produce, script) fire the existing multi-agent workflow, token-budgeted. No run should ever still be on auto-work at the next tick.
  3. Park and notify once.amber Any run that just landed on a gate: send one DingTalk line via the kao bot (Spark · GATE 2 open · <topic> · <url>). Dedupe per gate so it never nags twice for the same decision.
  4. Feed the front. If runs-parked-at-a-gate < R: pull the top Audience_Fit greenlit spark, spawn a fresh /spark run, drive it to GATE 1, notify. Repeat until R is met or the budget is spent.
  5. Top up the lake. If greenlit-ready < B: trigger an extra /spark_scan beyond the Monday cron. Otherwise leave it to the heartbeat.
  6. Budget guard. Cap tokens per tick. A research+script workflow runs ~1.3M; a live voice pick ran ~180k per script before you cut it. If the next action would blow the cap, defer it to the next tick, log it, and ping only if it is blocking you.
  7. Pick the next wake. Workflow still running, self-schedule ~270s. Everything parked at gates and buffers full, sleep to the next heartbeat. Nothing pending at all, push the fallback wake long.

06 The autonomy dial

The one real choice. Gates are human by default, but your run.json already carries an autonomous_run flag, and the shopify run proved it works end to end. So gate-crossing is a per-run switch, not a global mode.

Default · gated

The loop stages, you curate

Controller drives every auto leg and parks at all four gates. It kills the "/spark next" nudging and the cold start, but you still pick every angle and every voice. Whenever you sit down, a decision is already waiting with the work staged behind it.

GATE 1 you pick angles GATE 2 you pick winners GATE 3/4 you pick voice
Opt-in · autonomous

Set the flag, walk away

Mark a run autonomous_run:true and the controller self-clears its gates: GATE 1 takes top-K by Audience_Fit per pillar, GATE 2 keeps the researched winners, GATE 3/4 default to the bank-only fast polish (not the 2M-token live path). End to end, no stops.

use for a proven pillar keep gated for new territory
This is the answer to "loop or goal". The loop is the heartbeat trigger. The goal is the buffers plus zero dead air. The dial is how much of your judgement the goal is allowed to substitute. They were never rivals: the loop is the container, the goal is the contents, the dial is the lid.

07 Worked example · real state

Starting from the pipe as it actually is right now: shopify-get-paid is done, _active points at it, the bank holds 33 raw scan sparks plus a few moodboard and steven_input entries, almost nothing greenlit.

Tick 1 · Mon 11:40 (after the 09:40 weekly scan)
Cold pipe, so the loop fills it

shopify is done, retire it from active. Runs at a gate: 0, below R=3. Greenlit below B. Controller sharpens the top-fit raw sparks to greenlit, spawns run A on the strongest angle, builds the idea board, parks GATE 1, pings you. Budget left, so it stages B and C the same way. Sleeps to the next heartbeat.

Tick 2 · 13:40 (you cleared GATE 1 on A over lunch)
A advances, the loop catches it

A is now stage=listen, gate=null, so it gets drained: fire the research+produce workflow (the same 17-agent pattern as the shopify run, budgeted ~1.3M), build the cards page, park GATE 2, ping. B and C are still at GATE 1 and already notified, so no re-nag. Workflow is mid-flight, so it self-schedules a 270s re-check.

Tick 2b · +5 min (self-scheduled)
Workflow lands

Research and cards done, GATE 2 parked and pinged. Nothing else moving, drop back to the slow heartbeat.

Tick 3 · 15:40 (you flagged B autonomous)
B runs itself to done

You set autonomous_run:true on B and said "just ship that pillar". Controller self-clears B's gates: 3 angles per pillar by fit, research+scripts workflow, bank-only voice polish, final Google Doc assembled, logged to inbox, bank entries flipped to in_production then shipped. Zero stops. A is still parked at GATE 2, C still at GATE 1.

Net effect: whenever you open the laptop, there is always a gate open with the work already staged behind it, and the only thing the pipe ever waits on is your decision, or nothing at all when you have flagged a run autonomous.

08 What to actually build

Three small things. None of them touch the bank schema, /spark_scan, or the sub-skills. The loop only reads state and calls existing skills.

  1. A scheduled task spark-loopcron 0 9-21/2 * * * whose prompt is "run one Spark Loop reconcile tick" (the step-05 list). Mirrors the other scheduled-task SKILLs you already run.
  2. A reconcile reader Pipelines/Spark/scripts/loop_reconcile.py that loads bank + all run.json, returns a plan: which runs to drain, which gates to notify, how many new runs to spawn, B/R gaps. Pure state, no side effects, so a tick is cheap and testable.
  3. Two new run.json fields: last_notified_gate (dedupe the pings) and budget_spent (per-run token tally for the guard). Both additive, nothing existing breaks.

Default the dial to gated, B=10, R=3, heartbeat 2h across 09–21. Tune R up once you trust it. The first version can even skip the workflow-firing and only do steps 1, 3, 4, 7 (stage boards to GATE 1 and notify), which already removes the cold start and the nudging, then add the drain side once the parking behaviour feels right.
